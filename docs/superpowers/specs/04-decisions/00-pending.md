## 决策 1: Endpoint 命名规则

### 选项 A: 语义命名（推荐）

```
in:user, out:llm, in:tool, out:error
```

**优点**:
- 直观、可读性好
- 名称即语义
- 适合大型项目

**缺点**:
- 命名需要规范
- 可能冲突

---

### 选项 B: 简单序号

```
in_1, in_2, out_1, out_2
```

**优点**:
- 简单、不易冲突
- 框架自动管理

**缺点**:
- 可读性差
- 无法从名称推断用途

---

### 选项 C: 复合命名

```
in.user, out.llm, in.tool.result, out.agent.response
```

**优点**:
- 兼顾灵活性和可读性
- 支持层级结构
- 可分组管理

**缺点**:
- 实现复杂度中等

---

## 决策 2: 边的连接方式

### 选项 A: 显式端点连接

```
A.out:llm -> B.in:tool
A.out:error -> C.in:handler
```

**优点**:
- 精确控制
- 显式声明，无歧义
- 易于调试

**缺点**:
- 连接语句较长
- 修改麻烦

---

### 选项 B: 端点自动匹配

```
A -> B  // 自动匹配同名端点
// 框架自动将 A 的 out:* 连接到 B 的同名 in:*
```

**优点**:
- 简化连接语句
- 减少 boilerplate

**缺点**:
- 隐式行为，可能产生意外匹配
- 调试困难

---

### 选项 C: 端口组 + 广播

```
A.out:events -> B.in:*  // 广播到 B 的所有输入端点
A.out:llm -> B.in:tool, C.in:request  // 多目标
```

**优点**:
- 支持一对多
- 灵活表达复杂拓扑

**缺点**:
- 行为复杂
- 需要清晰的组管理

---

## 决策 3: 消息路由规则

### 选项 A: 精确路由

```
消息只发送到指定的 Endpoint
A.out:llm -> B.in:tool  // 只有 B.in:tool 收到
```

**优点**:
- 精确控制
- 可预测性强

**缺点**:
- 灵活性较低

---

### 选项 B: 广播路由

```
消息发送到所有匹配的 Endpoint
A.out:events -> *  // 所有订阅的节点都收到
```

**优点**:
- 适合事件通知场景
- 解耦发送方和接收方

**缺点**:
- 可能产生大量消息
- 难以追踪

---

### 选项 C: 条件路由

```
根据消息内容决定路由目标
if (msg.payload.type === 'tool') -> B.in:tool
else -> C.in:default
```

**优点**:
- 动态路由
- 适合复杂逻辑

**缺点**:
- 路由逻辑复杂
- 难以静态分析

---

### 选项 D: 通配符模式

```
A.out:* -> B.in:*  // 所有从 A 来的消息都发送到 B
```

**优点**:
- 简洁
- 适合测试/快速原型

**缺点**:
- 过于宽泛
- 难以精确控制

---

### 决策问题

**请选择**: A / B / C / D / 组合（请说明）？

**是否需要支持多种路由规则？**

## 决策 4: 类型安全设计方案

### 选项 A: 泛型约束（推荐）

```typescript
interface Endpoints {
  [name: string]: EndpointSlot<any>;
}

interface TypedEndpoints<In extends Endpoints, Out extends Endpoints> {
  in: In;
  out: Out;
}

interface Node<In extends Endpoints, Out extends Endpoints> {
  handle(endpoint: keyof In & string, msg: In[keyof In]): Promise<Out[keyof Out]>;
}

// 使用示例
interface LLMNodeEndpoints extends TypedEndpoints<
  { user: Slot<UserMsg>; tool: Slot<ToolResult> },
  { llm: Slot<LLMRequest>; error: Slot<ErrorMsg> }
> {}
```

**优点**:
- 编译时检查
- IDE 自动补全
- 类型推断

**缺点**:
- TypeScript 复杂性
- 可能有性能开销

---

### 选项 B: 接口定义 + 运行时校验

```typescript
interface LLMNodeEndpoints {
  in: {
    user: { type: 'user'; schema: UserMsgSchema };
    tool: { type: 'tool'; schema: ToolResultSchema };
  };
  out: {
    llm: { type: 'llm'; schema: LLMRequestSchema };
    error: { type: 'error'; schema: ErrorMsgSchema };
  };
}
```

**优点**:
- 显式声明
- 运行时可校验
- 文档化

**缺点**:
- 冗长
- 灵活性较低

---

### 选项 C: 简单联合类型

```typescript
type Endpoint = {
  name: string;
  payload: unknown;
};

// 消息带上端点信息
interface TypedMessage<T = unknown> {
  endpoint: string;
  payload: T;
}
```

**优点**:
- 简单
- 灵活

**缺点**:
- 类型安全弱
- 需要手动类型转换

## 决策 5: 高级特性

|------|------|
| **不支持** | 简单实现，缓冲区满则丢弃或阻塞 |
| **背压** | 上游节点自动降速 |
| **信号控制** | 通过特殊消息通知流控状态 |

**是否需要**: 否 / 背压 / 信号控制？

---

### 5.2 优先级队列

| 选项 | 说明 |
|------|------|
| **不支持** | FIFO 顺序 |
| **消息优先级** | 某些消息可优先处理 |
| **端点优先级** | 某些端点优先被处理 |

**是否需要**: 否 / 消息优先级 / 端点优先级？

| 选项 | 说明 |

| **不支持** | 抛出错误 |
| **丢弃** | 静默丢弃 |
| **特殊端点** | 路由到专门的 `in:dlq` 端点 |

**是否需要**: 否 / 丢弃 / DLQ？

### 5.4 消息转换/适配
